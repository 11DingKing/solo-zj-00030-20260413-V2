### 3.3 Alembic vs create_all 对比

| 特性 | Alembic | create_all |
|------|---------|------------|
| **版本控制** | ✅ 完整的迁移历史 | ❌ 无版本记录 |
| **回滚能力** | ✅ 支持 downgrade | ❌ 无法回滚 |
| **增量迁移** | ✅ 只执行变更部分 | ❌ 全量创建 |
| **数据迁移** | ✅ 支持数据转换 | ❌ 仅结构创建 |
| **团队协作** | ✅ 迁移文件可版本控制 | ❌ 难以同步 |
| **生产环境** | ✅ 安全可控 | ❌ 风险较高 |

### 3.4 使用 Alembic 的优势

1. **版本控制**:
   - 每个数据库结构变更都有对应的迁移文件
   - 迁移文件可以纳入 Git 版本控制
   - 可以查看数据库结构的完整变更历史

2. **回滚能力**:
   - 每个迁移文件都有 `upgrade()` 和 `downgrade()` 方法
   - 如果迁移出现问题，可以回滚到之前的版本
   - 例如: `alembic downgrade -1` 回滚到上一个版本

3. **增量迁移**:
   - 只执行新增的迁移，而不是重新创建所有表
   - 保护现有数据不被丢失
   - 提高迁移效率

4. **团队协作**:
   - 多个开发者可以同时创建迁移文件
   - 通过版本控制同步数据库结构变更
   - 避免数据库结构不一致的问题

5. **生产环境安全**:
   - 可以在执行迁移前审查迁移脚本
   - 支持事务性迁移（部分数据库）
   - 可以在测试环境先验证迁移

### 3.5 为什么不直接使用 create_all

1. **数据丢失风险**:
   - `create_all()` 会创建所有不存在的表，但不会修改已存在的表
   - 如果需要修改表结构，`create_all()` 无法处理
   - 如果使用 `drop_all()` 然后 `create_all()`，会丢失所有数据

2. **无法追踪变更**:
   - 没有历史记录，不知道数据库结构何时发生了什么变化
   - 无法回滚到之前的结构版本

3. **团队协作困难**:
   - 多个开发者修改模型时，难以同步数据库结构
   - 容易出现"在我机器上能运行"的问题

4. **生产环境不适用**:
   - 生产环境有大量数据，不能随意删除和重建表
   - 需要可控的、增量的迁移方式

### 4.4 Worker 配置对高并发的影响

#### 4.4.1 Worker 数量的影响

**Gunicorn 的工作模型**:
- Gunicorn 使用预派生（pre-fork）工作模型
- 主进程启动时创建指定数量的 worker 进程
- 每个 worker 进程独立处理请求
- 主进程负责监控和管理 worker 进程

**Worker 数量与并发能力**:
- **理论上**: 更多的 worker 可以处理更多的并发请求
- **实际上**: 受限于CPU、内存和I/O能力

**不同场景的影响**:

1. **CPU密集型任务**:
   - 最佳实践: `workers = 2 * CPU核心数 + 1`
   - 原因: 每个worker在等待I/O时，其他worker可以利用CPU
   - 本项目默认: `workers = CPU核心数`，可能偏保守

2. **I/O密集型任务**:
   - 本项目的价格查询属于I/O密集型（网络请求、数据库查询）
   - 可以适当增加worker数量
   - 或者使用异步worker（如 `gevent` 或 `eventlet`）

3. **内存限制**:
   - 每个worker进程都会加载整个应用程序
   - 更多的worker = 更多的内存消耗
   - 需要根据服务器内存调整worker数量

**Worker 数量过多的问题**:
- **上下文切换开销**: CPU需要在多个进程间切换，增加开销
- **内存压力**: 每个worker都有独立的内存空间，过多会导致内存不足
- **竞争加剧**: 多个worker竞争数据库连接、文件句柄等资源

**Worker 数量过少的问题**:
- **CPU利用率低**: 无法充分利用多核CPU
- **请求排队**: 并发请求多时，请求会在队列中等待
- **响应延迟**: 单个worker处理慢时，影响整体响应时间

#### 4.4.2 Keepalive 配置的影响

**配置值**: `keepalive = 120`（120秒）

**作用**:
- 保持客户端与服务器之间的TCP连接活跃
- 允许在同一个连接上发送多个HTTP请求
- 减少TCP握手和挥手的开销

**对高并发的影响**:

**优点**:
1. **减少连接开销**: 避免频繁建立和关闭TCP连接
2. **降低延迟**: 复用连接，减少握手时间
3. **提高吞吐量**: 相同时间内可以处理更多请求

**缺点**:
1. **资源占用**: 保持连接需要占用服务器资源（文件句柄、内存）
2. **连接耗尽**: 高并发时，可能导致可用连接耗尽
3. **负载均衡问题**: 长时间保持连接可能导致负载不均衡

**适用场景**:
- 120秒的keepalive时间适合:
  - 客户端会发送多个连续请求的场景
  - 网络延迟较高的环境
  - 需要减少连接开销的场景

#### 4.4.3 其他配置的影响

**Bind 配置**:
- 默认绑定到 `0.0.0.0:8000`
- 允许从任何网络接口访问
- 生产环境可能需要更严格的绑定

**日志配置**:
- 默认日志级别为 `info`
- 错误日志输出到标准错误
- 高并发场景下，合理的日志配置可以帮助排查问题，但过多日志会影响性能

### 4.5 高并发场景的优化建议

基于本项目的特点（价格查询是I/O密集型操作），建议:

1. **调整 Worker 数量**:
   - 对于I/O密集型任务，可以考虑:
     - 增加 `WORKERS_PER_CORE` 到 2-4
     - 或者使用异步worker类型
   - 例如: `WORKERS_PER_CORE=2`，4核CPU时worker数量为8

2. **考虑使用异步 Worker**:
   - Gunicorn 支持多种worker类型:
     - `sync`: 默认，同步worker
     - `gevent`: 基于协程的异步worker
     - `eventlet`: 另一种协程实现
   - 对于I/O密集型任务，`gevent` 可以显著提高并发能力
   - 需要安装: `pip install gevent`
   - 启动时指定: `gunicorn -k gevent ...`

3. **调整 Keepalive 时间**:
   - 根据实际访问模式调整
   - 如果客户端是短连接为主，可以适当缩短
   - 如果是API服务，保持较长时间更有利

4. **结合反向代理**:
   - 生产环境通常使用 Nginx 作为反向代理
   - Nginx 可以处理静态文件、SSL终止、负载均衡
   - 让 Gunicorn 专注于处理动态请求

5. **监控和调优**:
   - 监控服务器资源使用情况（CPU、内存、网络）
   - 根据实际负载调整配置
   - 没有"最佳"配置，只有"最适合"的配置

### 4.6 本项目配置分析

**当前配置的特点**:
- Worker数量: 基于CPU核心数，每个核心1个worker
- Keepalive: 120秒，较长的保持时间
- 绑定地址: 0.0.0.0，允许外部访问

**适用场景**:
- 适合中小型部署
- 对于I/O密集型的价格查询，可能需要更多worker
- 120秒的keepalive适合API服务场景

**潜在优化点**:
- 考虑增加worker数量或使用异步worker
- 根据实际访问模式调整keepalive时间
- 生产环境结合Nginx反向代理

## 5. 总结

### 5.1 价格数据流转

- **实时爬取**: 价格数据不存入数据库，而是实时爬取
- **完整链路**: 前端 → API → 后端服务 → 实时爬取 → 前端展示
- **关键技术**: `httpx` 异步HTTP请求、`selectorlib` 页面提取、`asyncio` 并发处理

### 5.2 数据库设计

- **无Price模型**: 价格数据实时爬取，不持久化
- **Shop模型**: 存储爬取配置信息
- **多对多关系**: `UserShops` 关联表实现用户和商店的多对多关系
- **灵活配置**: `listing_page_selector` 使用JSON格式，支持复杂的页面提取规则

### 5.3 Alembic 迁移

- **版本控制**: 完整的迁移历史，可追踪可回滚
- **团队协作**: 迁移文件可版本控制，便于团队同步
- **生产安全**: 增量迁移，保护现有数据
- **对比create_all**: 提供了更强大、更安全的数据库结构管理能力

### 5.4 Gunicorn 配置

- **Worker计算**: 基于CPU核心数，可通过环境变量调整
- **并发影响**: Worker数量直接影响并发处理能力，但需平衡资源消耗
- **Keepalive**: 120秒的保持时间适合API服务，减少连接开销
- **优化建议**: 对于I/O密集型任务，考虑增加worker或使用异步worker
